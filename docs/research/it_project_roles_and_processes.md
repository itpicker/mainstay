# IT IT 프로젝트 표준 인력 구성 및 업무 프로세스

이 문서는 IT 프로젝트 진행 시 일반적으로 투입되는 인력의 구성, 역할(R&R), 그리고 표준적인 업무 프로세스(Agile 및 Waterfall)를 정리한 것입니다.

## 1. 프로젝트 인력 구성 및 역할 (Roles & Responsibilities)

성공적인 IT 프로젝트를 위해 필요한 핵심 역할과 그들의 책임은 다음과 같습니다. 프로젝트 규모와 특성에 따라 한 사람이 여러 역할을 수행하거나, 한 역할에 여러 명이 투입될 수 있습니다.

### 핵심 관리 및 기획 직군
*   **Project Sponsor (프로젝트 스폰서)**
    *   **정의:** 프로젝트의 실질적인 물주이자 경영진 수준의 후원자입니다.
    *   **역할:** 예산 승인, 핵심 의사결정, 프로젝트의 비지니스 가치 보장, 조직 내 정치적 장애물 제거.
*   **Project Manager (PM, 프로젝트 관리자)**
    *   **정의:** 프로젝트의 시작부터 종료까지 전체를 총괄하는 책임자입니다.
    *   **역할:** 일정/범위/비용 관리, 리스크 관리, 인력 소싱 및 할당, 이해관계자 커뮤니케이션, 이슈 해결.
*   **Product Owner (PO, 프로덕트 오너) - 주로 Agile**
    *   **정의:** 제품의 '무엇'을 만들지를 결정하는 제품 책임자입니다.
    *   **역할:** 제품 백로그(Backlog) 관리 및 우선순위 결정, 요구사항 정의, 결과물(Increment) 수용/반려 결정.
*   **Business Analyst (BA, 비즈니스 분석가) / 기획자**
    *   **정의:** 비즈니스 요구사항을 기술적 요구사항으로 변환하는 가교 역할을 합니다.
    *   **역할:** 요구사항 수집 및 분석, 화면 설계(Wireframe/SB), 기능 명세서 작성, 프로세스 정의.

### 개발 및 기술 직군
*   **Project Leader (PL, 프로젝트 리더)**
    *   **정의:** 개발팀 또는 특정 파트의 기술적 리더입니다.
    *   **역할:** 기술적 의사결정, 코드 리뷰, 개발 일정 상세 관리, 개발팀 멘토링, 아키텍처 설계 지원.
*   **Software Architect (SW 아키텍트)**
    *   **정의:** 시스템의 전체적인 구조를 설계하는 기술 전문가입니다.
    *   **역할:** 기술 스택 선정, 시스템 구조 설계, 성능/보안/확장성 고려, 기술 표준 수립.
*   **Software Developer (개발자 - Front-end, Back-end, Embedded 등)**
    *   **정의:** 실제 소프트웨어를 구현하는 담당자입니다.
    *   **역할:** 기능 구현, 단위 테스트 작성, 버그 수정, 기술 문서 작성.
*   **Database Administrator (DBA)**
    *   **정의:** 데이터베이스를 설계하고 운영/관리하는 전문가입니다.
    *   **역할:** DB 모델링, 성능 튜닝, 데이터 무결성 보장, 백업 및 복구 계획 수립.

### 디자인 및 품질 직군
*   **UI/UX Designer (디자이너)**
    *   **정의:** 사용자의 경험을 설계하고 시각적인 화면을 제작합니다.
    *   **역할:** 사용자 리서치, UI 디자인, 프로토타이핑, 디자인 가이드 수립, 그래픽 에셋 제작.
*   **Quality Assurance (QA, 품질 보증) / Tester**
    *   **정의:** 소프트웨어의 결함을 찾고 품질을 검증하는 담당자입니다.
    *   **역할:** 테스트 시나리오/케이스 작성, 기능/비기능 테스트 수행, 결함 보고 및 추적, 품질 리포트 작성.

### 지원 직군
*   **Scrum Master (스크럼 마스터) - Agile**
    *   **정의:** 애자일/스크럼 프로세스가 잘 지켜지도록 돕는 퍼실리테이터(Facilitator)입니다.
    *   **역할:** 스크럼 이벤트 진행(데일리 미팅 등), 장애물(Blocker) 제거, 팀의 자율성 증진 지원.
*   **DevOps Engineer**
    *   **정의:** 개발과 운영의 가교 역할을 하며 배포 및 인프라를 자동화합니다.
    *   **역할:** CI/CD 파이프라인 구축, 클라우드 인프라 관리, 모니터링 시스템 구축.

## 2. IT 프로젝트 업무 프로세스

프로젝트의 성격(불확실성, 규모 등)에 따라 적절한 방법론을 선택해야 합니다. 대표적으로 **Waterfall(폭포수)** 모델과 **Agile(애자일)** 모델이 있습니다.

### A. Waterfall (폭포수) 모델
단계별로 순차적으로 진행하며, 전 단계가 완료되어야 다음 단계로 넘어가는 전통적인 방식입니다. 요구사항이 명확하고 변경이 적은 프로젝트에 적합합니다.

1.  **요구사항 분석 (Requirements Analysis)**
    *   고객의 요구사항을 수집하고 상세하게 정리합니다. (요구사항 정의서)
2.  **설계 (Design)**
    *   요구사항을 바탕으로 시스템 구조, DB, 화면 등을 설계합니다. (화면 설계서, 아키텍처 설계서, ERD)
3.  **구현 (Implementation/Coding)**
    *   설계서를 바탕으로 실제 코드를 작성합니다.
4.  **테스트 (Testing)**
    *   구현된 시스템이 요구사항을 충족하는지 검증합니다. (단위/통합/시스템/인수 테스트)
5.  **배포 (Deployment)**
    *   시스템을 실제 운영 환경에 설치하고 오픈합니다.
6.  **유지보수 (Maintenance)**
    *   운영 중 발생하는 오류를 수정하고 기능을 개선합니다.

### B. Agile (애자일) 모델 (Scrum 중심)
짧은 주기(Sprint)로 개발을 반복하며, 변화하는 요구사항에 유연하게 대처하는 방식입니다. 불확실성이 높거나 빠른 시장 출시가 필요한 프로젝트에 적합합니다.

1.  **제품 백로그 구성 (Product Backlog)**
    *   개발해야 할 모든 기능(User Story)을 목록화하고 우선순위를 매깁니다.
2.  **스프린트 계획 (Sprint Planning)**
    *   이번 주기(보통 2~4주)에 개발할 백로그 아이템을 선정하고 세부 계획을 수립합니다.
3.  **스프린트 실행 (Sprint / Daily Scrum)**
    *   개발을 진행하며 매일 짧은 미팅(Daily Scrum)을 통해 진행 상황과 장애물을 공유합니다.
4.  **스프린트 리뷰 (Sprint Review)**
    *   스프린트 종료 시 개발된 동작하는 소프트웨어(Increment)를 이해관계자에게 시연하고 피드백을 받습니다.
5.  **스프린트 회고 (Sprint Retrospective)**
    *   팀의 일하는 방식(프로세스)을 되돌아보고 개선점을 도출하여 다음 스프린트에 반영합니다.

## 3. 프로젝트 기본 룰 및 커뮤니케이션 가이드

성공적인 협업을 위해 팀원들이 지켜야 할 최소한의 규칙을 정의합니다.

### 커뮤니케이션 룰
*   **공식 채널 사용:** 업무 논의는 Slack, Teams, JIRA 등 지정된 공식 툴을 사용합니다.
*   **신속한 응답:** 업무 시간 내 메시지에는 가급적 빠른 시간(예: 2시간 이내) 내에 응답하거나 확인 여부를 알립니다.
*   **이슈의 조기 공유:** 일정 지연이나 기술적 이슈 발생 시, 혼자 해결하려 하지 말고 즉시 리더나 팀에게 공유합니다. ("Bad news travels fast")
*   **회의록 작성:** 모든 공식 회의 후에는 결정 사항과 Action Item을 정리하여 공유합니다.

### 개발 및 문서화 룰
*   **코드 컨벤션 준수:** 팀에서 합의된 코딩 스타일 가이드(Lint 등)를 엄격히 준수합니다.
*   **Git 브랜치 전략:** Git Flow나 GitHub Flow 등 정의된 브랜치 전략을 따르며, PR(Pull Request)을 통해 코드 리뷰를 거친 후 병合(Merge)합니다.
*   **문서의 현행화:** 기획서, 설계서, API 문서 등은 코드 변경 시점에 맞춰 최신 상태로 유지합니다.
*   **테스트 필수:** 모든 주요 기능은 배포 전 반드시 테스트 코드 통과 또는 QA 승인을 거쳐야 합니다.

### 일정 관리
*   **진척률 관리:** JIRA, Trello 등의 도구를 사용하여 자신의 업무 상태(To Do, In Progress, Done)를 매일 최신화합니다.
*   **일정 준수:** 맡은 업무의 마감 기한을 준수하며, 불가피한 경우 사전에 조정 요청을 합니다.
