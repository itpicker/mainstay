# IT 프로젝트 단계별/분야별 산출물 리스트 (Project Deliverables List)

이 문서는 IT 프로젝트의 각 단계(Phase)와 역할(Role)별로 산출해야 하는 주요 문서 및 결과물(Artifacts)을 정의합니다. **Waterfall**과 **Agile** 방법론을 모두 고려하여 구성하였으며, 각 산출물의 발행 시점과 포함되어야 할 주요 항목을 명시합니다.

---

## 🏗️ 1. 착수 및 분석 단계 (Initiation & Analysis)
프로젝트의 목표를 설정하고, 비즈니스 및 기술적 요구사항을 상세화하는 단계입니다.

| 분야 (Domain) | 산출물 (Deliverable) | 발행 시점 (When) | 주요 항목 (Contents) | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **PM / 기획** | **프로젝트 수행 계획서**<br>(Project Plan) | 착수 보고 시 | 프로젝트 범위, 일정(WBS), 조직도, 인력 계획, 리스크 관리 계획, 의사소통 계획 | 필수 |
| **PM / 기획** | **요구사항 정의서**<br>(Requirements Spec) | 인터뷰 및 분석 후 | 기능/비기능 요구사항 ID, 요구사항 명, 상세 설명, 우선순위, 수용 여부 | Agile의 Backlog |
| **PM / 기획** | **용어 사전**<br>(Glossary) | 분석 단계 중 | 업무 용어, 약어, 영문명, 데이터 타입 (DB 컬럼명 표준화용) | |
| **디자인** | **디자인 컨셉 시안**<br>(Design Concept) | 분석 말기 | 메인 페이지 시안, 서브 페이지 스타일 가이드, 컬러 팔레트, 타이포그래피 | |
| **개발 (Arch)** | **기술 아키텍처 정의서**<br>(System Architecture) | 분석 말기 ~ 설계 초 | 하드웨어/소프트웨어 구성도, 기술 스택, 인터페이스 방식, 배포 아키텍처 | |

## 📐 2. 설계 단계 (Design)
분석된 요구사항을 바탕으로 실제 구현 가능한 형태로 구체화하는 단계입니다.

| 분야 (Domain) | 산출물 (Deliverable) | 발행 시점 (When) | 주요 항목 (Contents) | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **기획 / 디자인** | **화면 설계서 (SB)**<br>(Storyboard / Wireframe) | 설계 단계 | 화면 UI 레이아웃, 기능 상세 명세, 프로세스 흐름(FlowChart), 예외 처리 | Figma/PPT |
| **기획 / 디자인** | **디자인 가이드**<br>(Style Guide) | 디자인 확정 시 | 공통 컴포넌트(버튼, 입력폼 등) 스타일, 아이콘, 여백/간격 규정 | Pub/Dev 가이드 |
| **개발 (DB)** | **데이터모델링 정의서(ERD)**<br>(Entity Relationship Diagram) | 설계 단계 | 논리/물리 모델링, 테이블 정의, 컬럼 정의, 관계(FK) 설정, 인덱스 설계 | |
| **개발 (DB)** | **테이블 정의서** | 설계 말기 | 테이블 목록, 컬럼 속성(Type, Length), Null 여부, 코멘트 | 엑셀/위키 |
| **개발** | **API 명세서**<br>(API Specification) | 설계 말기 | API 목록, Request/Response 파라미터, URL, Method, 에러 코드 | Swagger 활용 |
| **개발** | **프로그램 목록**<br>(Program List) | 구현 착수 전 | 구현 대상 프로그램ID, 프로그램명, 난이도, 담당자 | 진행율 관리용 |
| **QA** | **테스트 계획서**<br>(Test Plan) | 설계 말기 | 테스트 대상 및 범위, 수행 일정, 환경, 테스트 전략, 단계별 통과 기준 | |

## 💻 3. 구현 단계 (Implementation)
설계 산출물을 바탕으로 실제 코드를 작성하고 단위 테스트를 수행하는 단계입니다.

| 분야 (Domain) | 산출물 (Deliverable) | 발행 시점 (When) | 주요 항목 (Contents) | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **개발** | **소스 코드**<br>(Source Code) | 수시 (Git Commit) | 형상 관리 시스템(Git)에 등록된 소스 코드 파일 | |
| **개발** | **단위 테스트 결과서**<br>(Unit Test Report) | 개발 완료 시 | 테스트 케이스 ID, 입력값, 예상 결과, 실제 결과, 성공/실패 여부 | 자동화 권장 |
| **개발** | **빌드/배포 스크립트** | 구현 중 | CI/CD 파이프라인 설정 파일, Dockerfile, 배포 가이드 | |
| **디자인/퍼블** | **퍼블리싱 산출물**<br>(HTML/CSS/JS) | 화면 구현 시 | HTML 파일, CSS 스타일시트, 이미지 에셋 | |
| **기획** | **메뉴 구조도 (IA) 현행화** | 구현 중 변경 시 | 최종 확정된 메뉴 트리 및 메뉴 아이디 | |

## 🧪 4. 테스트 단계 (Test & Validation)
구현된 시스템의 품질을 검증하고 결함을 제거하는 단계입니다.

| 분야 (Domain) | 산출물 (Deliverable) | 발행 시점 (When) | 주요 항목 (Contents) | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **QA** | **통합 테스트 시나리오/케이스**<br>(TC) | 테스트 수행 전 | 테스트 시나리오, 수행 절차(Step), 기대 결과, 테스트 데이터 | |
| **QA** | **통합 테스트 결과서**<br>(Test Result Report) | 테스트 완료 시 | 케이스별 수행 결과(Pass/Fail), 결함 리스트, 결함 조치율, 잔존 결함 | |
| **기획 / QA** | **결함(Bug) 리포트** | 테스트 수행 중 | 버그 발생 경로, 재현 방법, 심각도, 조치 상태 (JIRA 등 이슈 트래커 활용) | |
| **PM** | **성능 테스트 결과서**<br>(Performance Report) | 오픈 준비 시 | 동시 접속자 수, TPS, 응답 속도, 서버 자원(CPU/Memory) 사용률 | 필요 시 |
| **사용자** | **UAT(인수 테스트) 결과서** | 오픈 직전 | 현업 담당자의 테스트 수행 결과 및 최종 승인 서명 | 오픈 허들 |

| 분야 (Domain) | 산출물 (Deliverable) | 발행 시점 (When) | 주요 항목 (Contents) | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **PM / 기획** | **사용자 매뉴얼**<br>(User Guide) | 오픈 전 | 시스템 접속 방법, 메뉴별 사용법, 주요 기능 시나리오, FAQ | |
| **개발 / 운영** | **운영자 매뉴얼**<br>(Admin Guide) | 오픈 전 | 시스템 구동/중지 방법, 모니터링 방법, 백업/복구 절차, 관리자 메뉴 사용법 | |
| **개발** | **프로그램 명세서** | 종료 시 | 최종 소스 코드 설명, 클래스 다이어그램, 주요 로직 설명 | |

| **PM** | **검수 확인서** | 프로젝트 종료 | 과업 이행 여부 확인, 계약 목적물 인수 확인 (고객사 날인) | |

## 🔄 Agile (Scrum) 프로젝트의 주요 산출물

Agile 방법론에서는 위 산출물을 **간소화**하거나 **대체**하여 운영합니다. 문서는 최소화하고 동작하는 소프트웨어를 중시합니다.

1.  **Product Backlog:** 요구사항 정의서를 대체. 모든 기능과 개선사항의 우선순위화된 목록.
2.  **Sprint Backlog:** 해당 스프린트(2~4주)에 개발할 대상 목록.
3.  **User Story:** 기능 명세 상세. ("As a [User], I want to [Action], so that [Benefit]")
4.  **Burndown Chart:** 스프린트 진행 상황 시각화.
5.  **Working Software (Increment):** 각 스프린트 종료 시 동작 가능한 소프트웨어. (가장 중요한 산출물)
6.  **Sprint Retrospective Note:** 회고를 통해 도출된 개선 사항 기록.
